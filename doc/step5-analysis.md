# Step 5: LLM Analysis

The final pipeline step sends the prefiltered trace and focus-specific
documentation to an LLM, which produces a structured diagnostic report
following a fill-in-the-blank template.

**Source files:** `scripts/analyze.py`, `scripts/templates.py`
**Output:** `results/analyze.md`

## Documentation Loading

The `FOCUS_DOCS` dict maps focus area strings to BlueZ documentation
files in RST format:

| Focus Area | Documentation Files |
|------------|---------------------|
| Connection issues | `btmon-connections.rst` |
| Controller enumeration | `btmon-hci-init.rst` |
| Pairing / Security | `btmon-smp.rst` |
| GATT discovery | `btmon-gatt.rst` |
| Audio | `btmon-le-audio.rst`, `btmon-a2dp.rst`, `btmon-hfp.rst` |
| Audio / LE Audio | `btmon-le-audio.rst` |
| Audio / A2DP | `btmon-a2dp.rst` |
| Audio / HFP | `btmon-hfp.rst` |
| L2CAP channel issues | `btmon-l2cap.rst` |
| Advertising / Scanning | `btmon-advertising.rst` |
| Disconnection analysis | `btmon-connections.rst` |

`load_docs(docs_path, focus)` tries focus-specific files first.  If
they exist, only the relevant documentation is loaded (important for
the GitHub Models free tier where context is limited).  Falls back to
the full `btmon.rst` if focus-specific files are not found.

Documentation is then truncated via `truncate_for_context()` to fit
within the provider's docs character limit.

## Context Budgets

Each LLM provider has different context limits:

| Provider | Trace Limit | Docs Limit | Default Model |
|----------|:-----------:|:----------:|---------------|
| GitHub Models (free) | 16,000 chars | 4,000 chars | `openai/gpt-4o-mini` |
| OpenAI | 100,000 chars | 50,000 chars | `gpt-4o` |
| Anthropic | 100,000 chars | 50,000 chars | `claude-sonnet-4-20250514` |

The trace limit determines the `max_chars` passed to `prefilter()`.
The docs limit determines how much documentation text is included in
the system prompt.

## Prompt Construction

`build_prompt()` produces a `(system_prompt, user_prompt)` pair.

### System Prompt

```
You are a Bluetooth protocol analyst specializing in BlueZ btmon trace
analysis.  You have deep knowledge of HCI, L2CAP, ATT/GATT, SMP, and
LE Audio protocols.

<btmon-documentation>
{docs_text}
</btmon-documentation>

Your task is to analyze a decoded btsnoop trace and produce a
diagnostic report.  Be specific — reference actual handle values,
opcodes, error codes, and timestamps from the trace.
{clip_note}

Format your report in GitHub-flavored markdown.
```

The `{clip_note}` section is conditionally added based on trace type:

**For prefiltered traces** (annotator was used): Explains the
three-section structure (summary header, annotations, raw trace),
instructs the LLM to cross-reference annotations with raw trace via
frame numbers, and defines PASS criteria (all operations succeed,
graceful disconnects are normal, ISO data volume is expected).

**For clipped traces** (pattern-based, no annotator): Notes that the
trace was clipped around error patterns with context packets
preserved.

**When absence errors exist:** Appends the list of protocol-flow
issues as investigation hints.

### User Prompt

```
## Analysis Request

**User description:** {description}
**Focus area:** {focus}

## Decoded btsnoop trace

```
{decoded_text}
```

## Output format

{output_instructions}
```

The `{output_instructions}` are generated by `template_instructions()`
from `templates.py`.

## Template System

### Template Structure

Every template combines three parts:

1. **`_HEADER`** -- Common report header with focus area, auto-detection
   flag, verdict, and one-line summary in a metadata table.
2. **Body** -- Focus-specific sections with tables and field labels.
3. **`_FOOTER`** -- Recommendations section and attribution.

Audio-area templates share `_AUDIO_STREAMS` -- a common Audio Streams
table section.

### Available Templates

| Template Key | Focus Areas | Key Sections |
|-------------|-------------|--------------|
| `GENERAL` | General, GATT | Connection Timeline, Protocol Analysis, Issues |
| `A2DP` | Audio / A2DP | Timeline, Audio Streams, AVDTP Signaling, Issues |
| `HFP` | Audio / HFP | Timeline, RFCOMM Setup, SLC (AT Commands), Codec Negotiation, SCO Connection, Issues |
| `LE_AUDIO` | Audio / LE Audio | Timeline, Audio Streams, CIS/BIG Setup, Issues |
| `AUDIO` | Audio (combined) | Timeline, Audio Streams, Protocol Analysis, Issues |
| `CONNECTIONS` | Connection issues | Timeline (with handles/addresses), Active Connections, Disconnections, Issues |
| `SMP` | Pairing / Security | Timeline, Pairing Parameters (Initiator/Responder), Pairing Flow, Issues |
| `L2CAP` | L2CAP channel issues | Timeline, L2CAP Channels (PSM/CID/MTU), Signaling Issues, Issues |
| `ADVERTISING` | Advertising / Scanning | Timeline, Devices Seen, Advertising Config, Issues |
| `HCI_INIT` | Controller enumeration | Initialization Stages, Controller Identity, Buffer Config, Key Features, Issues |
| `DISCONNECTION` | Disconnection analysis | Timeline, Disconnection Events, Pre-Disconnection Activity, Issues |

### Template Placeholder Types

- **`{field}`** -- Single-value fields (e.g. `{verdict}`, `{sco_handle}`).
  LLM fills with a value extracted from the trace, or `N/A`.
- **`{..._rows}`** -- Multi-row table expansions (e.g. `{timeline_rows}`,
  `{avdtp_rows}`).  LLM generates as many rows as needed.
- **`{section}`** -- Free-text sections (e.g. `{protocol_analysis}`,
  `{issues}`, `{recommendations}`).

### Formatting Rules

`template_instructions()` generates 9 strict rules enforced on the LLM:

1. **Reproduce every heading and table exactly** -- no additions,
   removals, or reordering.
2. **Replace `{placeholder}` with values** -- use `N/A` if unknown.
3. **Expand `{..._rows}`** to as many rows as needed.
4. **Verdict** must be exactly `PASS`, `FAIL`, or `INCONCLUSIVE`:
   - `PASS` -- protocol flow completed without errors.  Graceful
     disconnects (0x13, 0x16) are normal.
   - `FAIL` -- actual errors, rejects, non-Success status codes,
     or unexpected disconnections.
   - `INCONCLUSIVE` -- incomplete or ambiguous trace.
5. **One-line summary** -- single sentence, max 120 characters.
6. **Issues Found format** -- each issue must have: Issue N title,
   Timestamp, What happened, Why it matters, Root cause.
   If no issues: `No issues found.`
7. **Recommendations** -- numbered list of concrete, actionable steps.
   If none: `No recommendations.`
8. **Audio Streams table** -- filled from `STREAM:` diagnostics.
   Direction `dir=?` may be inferred from context (append `(inferred)`).
9. **No text outside template** -- no preamble, closing remarks, or
   apologies.

## Focus Normalization

`normalize_focus(raw)` maps user-provided focus strings to canonical
keys used across the pipeline:

1. Exact match against `_KNOWN_FOCUS` set.
2. Case-insensitive alias match (e.g. `"a2dp"` -> `"Audio / A2DP"`).
3. Substring match against aliases (longest alias first).
4. Substring match against canonical keys.
5. Fallback: `"General (full analysis)"` with a warning.

This ensures the issue template dropdown values, free-text input, and
internal references all resolve to the same canonical key.

## LLM Provider Implementations

All three providers use the same prompt structure (`system` + `user`
messages), `max_tokens=4096`, and `temperature=0.2`.

### OpenAI

```
POST https://api.openai.com/v1/chat/completions
Authorization: Bearer {OPENAI_API_KEY}
```

### Anthropic

```
POST https://api.anthropic.com/v1/messages
x-api-key: {ANTHROPIC_API_KEY}
anthropic-version: 2023-06-01
```

Uses `system` field instead of a system message in the messages array.

### GitHub Models

```
POST https://models.github.ai/inference/chat/completions
Authorization: Bearer {GH_MODELS_TOKEN or GITHUB_TOKEN}
```

Requires a PAT with `models:read` scope.  Falls back to `GITHUB_TOKEN`
but warns if it lacks Models access (401/403).

## Post-Processing

After receiving the LLM response:

1. Strip `<output-template>` / `</output-template>` wrappers that some
   models echo back.
2. Prepend `## Step 5: LLM Analysis` heading when writing to
   `--output-dir`.
3. Write to `results/analyze.md`.

The GitHub Actions workflow appends an attribution footer when posting
as an issue comment.

## Pipeline Orchestration Summary

The `main()` function in `analyze.py` ties all steps together:

1. Download trace via URL, decode with `btmon -r`, optionally anonymize.
2. **Step 1:** Run `detect()` + `select_focus()`.  Write `detect.md`.
3. **Steps 2-4:** Run `annotate_trace()` then `prefilter()` with
   pre-annotated packets.  Write `filter.md`, `annotate.md`, `diagnose.md`.
4. **Step 5:** Load docs, build prompt, call LLM.  Write `analyze.md`.
5. All step outputs are posted as separate GitHub issue comments by
   the CI workflow.
